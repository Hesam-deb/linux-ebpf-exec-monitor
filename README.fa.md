# پایشگر اجرای پردازش‌های لینوکس با eBPF

> ببینید سیستم لینوکسی شما چه برنامه‌هایی را اجرا می‌کند؛ مستقیم از هسته و تقریباً زنده.

[Read in English](README.md)

این پروژهٔ آموزشی کوچک، اجرای موفق پردازش‌ها را با eBPF دریافت می‌کند و نتیجه را در یک داشبورد فارسی نمایش می‌دهد. اگر می‌خواهید tracepointهای لینوکس، BCC، انتقال رویداد با perf buffer و اتصال آن‌ها به یک برنامهٔ Flask را عملی یاد بگیرید، این پروژه نقطهٔ شروع ساده‌ای است.

## چه چیزی می‌بینید؟

با اجرای هر برنامه، اطلاعات زیر در داشبورد ظاهر می‌شود:

- نام دستور یا پردازش؛
- شناسهٔ پردازش یا PID؛
- زمان محلی ثبت رویداد؛
- تعداد کل رویدادها و دستورهای یکتا؛
- وضعیت سلامت پایشگر و ظرفیت حافظه.

صفحه هر ۲ ثانیه به‌روز می‌شود و ۱۰۰ رویداد جدید را در حافظه نگه می‌دارد.

## پروژه چگونه کار می‌کند؟

```text
tracepoint لینوکس: sched:sched_process_exec
                         │
                         ▼
                  برنامهٔ BCC/eBPF
                         │ perf buffer
                         ▼
                  بارگذار Python
                         │
                         ▼
             حافظهٔ موقت thread-safe
                         │
                         ▼
                  داشبورد فارسی Flask
```

## پیش‌نیازها

### محیط پشتیبانی‌شده

- Ubuntu 24.04 LTS یا یک توزیع جدید مبتنی بر Ubuntu/Debian
- هستهٔ لینوکس نسخهٔ 6 یا جدیدتر
- معماری x86_64 یا ARM64 که بسته‌های BCC نصب‌شده از آن پشتیبانی کنند
- اینترنت هنگام نصب
- دسترسی `sudo`

اسکریپت نصب، موارد زیر را با `apt` نصب می‌کند:

- `python3`، `python3-pip` و `python3-venv`
- `bpfcc-tools`، `libbpfcc-dev` و `python3-bpfcc`
- هدرهای مربوط به هستهٔ در حال اجرا
- Flask نسخهٔ 3 در محیط مجازی `.venv`

BCC عمداً از بسته‌های Ubuntu نصب می‌شود، نه PyPI؛ چون بخش بومی آن باید با کتابخانه‌های سیستم و ابزارهای هسته هماهنگ باشد.

> این پروژه مستقیماً روی Windows یا macOS اجرا نمی‌شود، چون برنامه‌ای را داخل هستهٔ لینوکس بارگذاری می‌کند. پیشنهاد ساده‌تر، استفاده از ماشین مجازی Ubuntu است. WSL فقط در صورتی مناسب است که هسته و پشتیبانی BCC آن درست پیکربندی شده باشد.

## دریافت پروژه

روش HTTPS:

```bash
git clone https://github.com/Hesam-deb/linux-ebpf-exec-monitor.git
cd linux-ebpf-exec-monitor
```

روش SSH، در صورتی که کلید SSH گیت‌هاب را تنظیم کرده‌اید:

```bash
git clone git@github.com:Hesam-deb/linux-ebpf-exec-monitor.git
cd linux-ebpf-exec-monitor
```

آدرس‌های مخزن:

- HTTPS: `https://github.com/Hesam-deb/linux-ebpf-exec-monitor.git`
- SSH: `git@github.com:Hesam-deb/linux-ebpf-exec-monitor.git`

## نصب

```bash
chmod +x scripts/install.sh scripts/run.sh
./scripts/install.sh
```

اسکریپت نصب، سیستم‌عامل و نسخهٔ هسته را بررسی می‌کند، بسته‌های سیستمی را نصب می‌کند، محیط `.venv` را می‌سازد و در پایان امکان import کردن BCC و Flask را آزمایش می‌کند.

## اجرا

پایشگر را اجرا کنید:

```bash
sudo ./scripts/run.sh
```

سپس این آدرس را در مرورگر باز کنید:

```text
http://127.0.0.1:5000
```

داشبورد به‌صورت پیش‌فرض فارسی است. با کلید تغییر زبان بالای صفحه می‌توانید نسخهٔ انگلیسی را ببینید یا مستقیماً این آدرس را باز کنید:

```text
http://127.0.0.1:5000/?lang=en
```

ترمینال اول را باز نگه دارید. در ترمینال دوم چند دستور اجرا کنید:

```bash
ls
whoami
ip a
ping -c 2 8.8.8.8
```

حداکثر تا ۲ ثانیه بعد، دستورهای جدید باید در داشبورد دیده شوند. برای توقف پایشگر، در ترمینال اول `Ctrl+C` را بزنید.

## تنظیم آدرس و پورت

داشبورد به‌صورت پیش‌فرض فقط روی localhost و پورت 5000 در دسترس است. برای تغییر آن‌ها:

```bash
sudo FLASK_HOST=0.0.0.0 FLASK_PORT=8080 ./scripts/run.sh
```

استفاده از `0.0.0.0` داشبورد را در شبکهٔ محلی قابل دسترس می‌کند. چون این نمونه احراز هویت ندارد، فقط در شبکهٔ قابل اعتماد از این حالت استفاده کنید.

خروجی JSON نیز در این آدرس موجود است:

```text
http://127.0.0.1:5000/api/events
```

## اجرای آزمون‌ها

آزمون‌ها eBPF را بارگذاری نمی‌کنند و به root نیاز ندارند:

```bash
.venv/bin/python -m unittest discover -s tests -v
```

```bash

```

## محدوده و محدودیت‌ها

این پروژه برای یادگیری است، نه جایگزین یک عامل امنیتی عملیاتی:

- فقط اجرای پردازش را ثبت می‌کند و فایل، شبکه یا ورود کاربران را پایش نمی‌کند؛
- با توقف برنامه، رویدادهای حافظه از بین می‌روند؛
- فقط ۱۰۰ رویداد جدید نگهداری می‌شود؛
- داشبورد احراز هویت و TLS ندارد؛
- طول نام دستور به محدودیت نام task در لینوکس وابسته است؛
- ذخیره‌سازی دائمی، غنی‌سازی، هشدار و محافظت در برابر دست‌کاری وجود ندارد.
