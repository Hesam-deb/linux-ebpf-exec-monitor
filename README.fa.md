# پایشگر چرخهٔ عمر پردازش‌های لینوکس با eBPF

> اجرای پردازش‌ها را از لحظهٔ شروع تا پایان، مستقیم از هستهٔ لینوکس مشاهده کنید.

[English README](README.md) · [دانلود گزارش PDF با فونت وزیرمتن](docs/project-report.pdf)

این پروژه یک نمونهٔ آموزشی برای یادگیری eBPF و مشاهده‌پذیری لینوکس است. پایشگر، اجرای موفق و خروج پردازش‌ها را از هسته دریافت می‌کند، اطلاعات تکمیلی را از `/proc` می‌خواند و نتیجه را در یک داشبورد زندهٔ فارسی و انگلیسی نمایش می‌دهد.

## شروع سریع

```bash
git clone https://github.com/Hesam-deb/linux-ebpf-exec-monitor.git
cd linux-ebpf-exec-monitor
chmod +x scripts/install.sh scripts/run.sh
./scripts/install.sh
sudo ./scripts/run.sh
```

سپس این آدرس را باز کنید:

```text
http://127.0.0.1:5000
```

برای توقف برنامه در ترمینال `Ctrl+C` را بزنید.

> اجرای پروژه به Linux، هدرهای هماهنگ با هسته و دسترسی root نیاز دارد. محیط اصلی پشتیبانی‌شده Ubuntu 24.04 است.

## داشبورد چه چیزی نشان می‌دهد؟

برای هر پردازش این اطلاعات نمایش داده می‌شود:

- نام دستور، PID و PID والد؛
- نام کاربر و UID؛
- مسیر فایل اجرایی و آرگومان‌ها؛
- وضعیت «در حال اجرا» یا «پایان‌یافته»؛
- زمان شروع، زمان پایان و مدت اجرا؛
- کد خروج یا سیگنال پایان.

داده‌ها هر یک ثانیه، بدون بارگذاری دوبارهٔ صفحه، به‌روز می‌شوند. با باز کردن «جزئیات پردازش»، پنل در بالاترین لایه باقی می‌ماند و پس‌زمینه محو می‌شود. انتخاب شما هنگام به‌روزرسانی زنده حفظ خواهد شد.

## بار خط لولهٔ eBPF

بخش «بار خط لولهٔ eBPF» وضعیت مسیر انتقال رویدادها را نشان می‌دهد:

- نرخ رویداد در بازهٔ متحرک ۱۰ ثانیه؛
- تعداد رویدادهای اجرا و خروج؛
- تعداد رویدادهای ازدست‌رفته در perf buffer؛
- وضعیت سلامت بافر.

این معیارها فشار خط لولهٔ رویداد را نشان می‌دهند؛ نه مصرف دقیق CPU دستورهای eBPF در هسته.

دو نمودار زنده در بالای داشبورد نیز این اطلاعات را نشان می‌دهند:

- مصرف CPU و حافظهٔ خود پایشگر؛
- مصرف CPU و حافظهٔ سیستم و میانگین بار یک‌دقیقه‌ای؛
- تاریخچهٔ متحرک ۶۰ نمونهٔ اخیر.

## معماری

```text
هستهٔ لینوکس
  ├─ sched:sched_process_exec
  └─ sched:sched_process_exit
              │
              ▼
        برنامهٔ BCC/eBPF
              │ perf buffer
              ▼
          بارگذار Python
  ├─ تبدیل UID به نام کاربر
  ├─ خواندن /proc/<pid>/exe
  ├─ خواندن /proc/<pid>/cmdline
  └─ تفسیر کد خروج یا سیگنال
              │
              ▼
      حافظهٔ thread-safe چرخهٔ عمر
              │
        ┌─────┴─────┐
        ▼           ▼
  داشبورد Flask    API خروجی JSON
```

## پیش‌نیازها

- Ubuntu 24.04 یا توزیع جدید مبتنی بر Ubuntu/Debian؛
- هستهٔ Linux نسخهٔ 6 یا جدیدتر؛
- معماری x86_64 یا ARM64 پشتیبانی‌شده توسط BCC؛
- اینترنت هنگام نصب؛
- دسترسی `sudo`.

اسکریپت نصب این بسته‌ها را آماده می‌کند:

- `python3`، `python3-pip` و `python3-venv`
- `bpfcc-tools`، `libbpfcc-dev` و `python3-bpfcc`
- هدرهای هستهٔ در حال اجرا
- Flask نسخهٔ 3 در محیط `.venv`

BCC از بسته‌های سیستم‌عامل نصب می‌شود، چون بخش بومی آن باید با کتابخانه‌ها و ابزارهای هسته هماهنگ باشد.

> Windows و macOS نمی‌توانند این پروژه را مستقیماً اجرا کنند. استفاده از ماشین مجازی Ubuntu، راه ساده‌تر و قابل‌اعتمادتر است.

## اجرا و آزمایش

ترمینال اول:

```bash
sudo ./scripts/run.sh
```

ترمینال دوم:

```bash
ls
whoami
ip a
sleep 3
false
```

رویدادها باید حدود یک ثانیه بعد در داشبورد دیده شوند. دستور `sleep 3` برای مشاهدهٔ تغییر وضعیت از «در حال اجرا» به «پایان‌یافته» مناسب است و دستور `false` نمونه‌ای از کد خروج غیرصفر ایجاد می‌کند.

## تنظیم آدرس و پورت

حالت پیش‌فرض فقط روی سیستم محلی در دسترس است:

```text
http://127.0.0.1:5000
```

برای تغییر آدرس یا پورت:

```bash
sudo FLASK_HOST=0.0.0.0 FLASK_PORT=8080 ./scripts/run.sh
```

استفاده از `0.0.0.0` داشبورد را در شبکهٔ محلی منتشر می‌کند. چون پروژه احراز هویت و TLS ندارد، فقط در شبکهٔ قابل اعتماد از این حالت استفاده کنید.

## منظور از «فعالیت پردازش» چیست؟

این نسخه چرخهٔ عمر پردازش را دنبال می‌کند: چه کسی آن را اجرا کرده، پردازش والد چیست، با چه آرگومان‌هایی اجرا شده، چه مدت فعال بوده و چگونه پایان یافته است.

فعالیت‌های داخلی مانند باز کردن فایل، اتصال شبکه، تخصیص حافظه و همهٔ system callها در این نسخه پایش نمی‌شوند. این قابلیت‌ها به probeهای جداگانه، فیلتر دقیق و ذخیره‌سازی پرظرفیت‌تر نیاز دارند.

مسیر فایل اجرایی و آرگومان‌ها بلافاصله پس از رویداد اجرا از `/proc/<pid>` خوانده می‌شوند. اگر پردازش بسیار سریع پایان یابد، ممکن است این دو مقدار در دسترس نباشند؛ بااین‌حال رویداد اصلی هسته ثبت می‌شود.

## محدودیت‌ها

- داده‌ها فقط در حافظه نگهداری می‌شوند و با توقف برنامه از بین می‌روند؛
- فقط ۱۰۰ پردازش جدید حفظ می‌شود؛
- داشبورد احراز هویت و TLS ندارد؛
- نام کوتاه دستور به محدودیت `TASK_COMM_LEN` در Linux وابسته است؛
- تکمیل اطلاعات از `/proc` برای پردازش‌های بسیار کوتاه best effort است؛
- فایل، شبکه، ورود کاربران، هشدار و محافظت در برابر دست‌کاری پایش نمی‌شوند.

- سرویس systemd؛
- احراز هویت و TLS برای استفاده در شبکه.
