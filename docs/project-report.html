1. Project objective

The project started as a small execution monitor that reported PID, command name, and timestamp. It has been expanded into a process lifecycle monitor with a bilingual Persian/English dashboard and a documented installation workflow.

2. What was implemented

AreaCompleted work
Kernel monitoringCapture successful process execution and process exit with sched_process_exec and sched_process_exit.
IdentityCapture PID, parent PID, UID, and resolve the local user name.
InvocationRead executable path and command-line arguments from /proc/<pid> on a best-effort basis.
LifecycleCorrelate start and exit events, calculate duration, and decode exit code or terminating signal.
DashboardResponsive dark UI, Persian and English languages, RTL/LTR layouts, live one-second polling, persistent expanded details, eBPF pipeline-load metrics, and JSON output.
DocumentationFriendly English and Persian READMEs with HTTPS/SSH clone addresses, requirements, installation, troubleshooting, and limitations.
QualityAutomated model and Flask tests for lifecycle correlation, serialization, translations, and API behavior.

4. Process details now available

  • PID and parent PID (PPID)
  • UID and user name
  • Command name, executable path, and arguments
  • Running or exited status
  • Start time, exit time, and duration
  • Exit code or termination signal

Live eBPF pipeline load

The dashboard reports a rolling event rate, exec/exit totals, lost perf-buffer events, and buffer health. These values measure event-pipeline pressure rather than claiming to measure exact kernel instruction CPU cost. Data updates without a page reload, so expanded process details remain open.

“What the process is doing” is intentionally scoped to lifecycle information. Monitoring file access, network traffic, or every system call would require additional probes, filtering, and higher-volume storage.

5. Running the project

git clone https://github.com/Hesam-deb/linux-ebpf-exec-monitor.git
cd linux-ebpf-exec-monitor
chmod +x scripts/install.sh scripts/run.sh
./scripts/install.sh
sudo ./scripts/run.sh

Persian dashboard: http://127.0.0.1:5000
English dashboard: http://127.0.0.1:5000/?lang=en
JSON API: http://127.0.0.1:5000/api/events

6. Requirements and constraints

  • Ubuntu 24.04 or a recent Ubuntu/Debian-based Linux distribution
  • Linux kernel 6.x or newer, matching kernel headers, BCC packages, Python 3, Flask 3.x, and root privileges
  • Windows and macOS require an Ubuntu virtual machine
  • Events are retained in memory and are lost when the application stops
  • /proc enrichment can miss very short-lived processes
  • The dashboard is educational and has no authentication or TLS

۱. هدف پروژه

پروژه ابتدا یک پایشگر ساده برای نمایش PID، نام دستور و زمان اجرا بود. اکنون به پایشگر چرخهٔ عمر پردازش تبدیل شده و داشبورد دوزبانهٔ فارسی/انگلیسی، راهنمای نصب و API خروجی JSON دارد.

۲. قابلیت‌های اضافه‌شده

  • ثبت اجرای موفق و خروج پردازش با دو tracepoint هستهٔ لینوکس
  • ثبت PID، PPID و UID و تبدیل UID به نام کاربر
  • خواندن مسیر فایل اجرایی و آرگومان‌ها از /proc
  • تطبیق رویداد شروع و پایان و محاسبهٔ مدت اجرا
  • نمایش کد خروج یا سیگنال پایان
  • داشبورد واکنش‌گرا با زبان فارسی و انگلیسی، حالت RTL/LTR و به‌روزرسانی زندهٔ یک‌ثانیه‌ای
  • جزئیات بازشونده برای هر پردازش و خروجی JSON
  • حفظ پنل جزئیات بازشده و نمایش نرخ رویداد، رویدادهای ازدست‌رفته و سلامت perf buffer
  • آزمون‌های خودکار برای مدل، API و رابط دوزبانه

۳. مسیر داده

رویدادهای اجرا و خروج در هسته ثبت و از perf buffer به Python ارسال می‌شوند. بارگذار Python اطلاعات کاربر و /proc را اضافه می‌کند. سپس حافظهٔ thread-safe رویداد شروع و پایان را با PID تطبیق می‌دهد و Flask نتیجه را در داشبورد و API نمایش می‌دهد.

۴. اطلاعات قابل مشاهده برای هر پردازش

  • PID و PID والد
  • نام کاربر و UID
  • نام دستور، مسیر فایل اجرایی و آرگومان‌ها
  • وضعیت در حال اجرا یا پایان‌یافته
  • زمان شروع، زمان پایان و مدت اجرا
  • کد خروج یا سیگنال پایان

بار زندهٔ خط لولهٔ eBPF

داشبورد نرخ متحرک رویدادها، مجموع رویدادهای اجرا و خروج، رویدادهای ازدست‌رفته و سلامت perf buffer را نشان می‌دهد. این معیارها فشار خط لولهٔ رویداد را اندازه می‌گیرند، نه مصرف دقیق CPU دستورهای eBPF در هسته. به‌روزرسانی بدون بارگذاری دوبارهٔ صفحه انجام می‌شود و جزئیات باز می‌مانند.

منظور از فعالیت پردازش در این نسخه، چرخهٔ عمر آن است. بررسی فایل‌ها، شبکه و همهٔ system callها به probeهای جداگانه و زیرساخت ذخیره‌سازی قوی‌تری نیاز دارد.

۶. محدودیت‌ها

  • پروژه آموزشی است و احراز هویت، TLS و ذخیره‌سازی دائمی ندارد.
  • اطلاعات /proc برای پردازش‌های بسیار کوتاه ممکن است از دست برود.
  • برای اجرا به Linux، BCC، هدرهای هسته و دسترسی root نیاز است.
  • پایش فایل و شبکه در این نسخه انجام نمی‌شود.
